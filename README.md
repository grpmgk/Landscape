# Лабораторная работа: Ландшафт с LOD и оптимизациями

## Общее описание проекта

Проект — это **рендерер ландшафта (террейна)** на DirectX 12. Ландшафт строится по **карте высот (heightmap)**, разбивается на **тайлы** через **квадро-дерево**, для каждого тайла выбирается **уровень детализации (LOD)** и применяется **Frustum Culling** для отсечения невидимых частей.

---

## Основные задачи и их реализация

### 1. Ландшафт с квадро-деревом для определения LOD

**Что это:** Квадро-дерево — структура данных, которая делит ландшафт на квадратные узлы. Каждый узел может делиться на 4 дочерних (NW, NE, SW, SE). Уровень детализации (LOD) выбирается в зависимости от расстояния до камеры: близко — больше деталей, далеко — меньше.

**Где реализовано:**
- `QuadTree.h` / `QuadTree.cpp` — класс `QuadTree` и структура `TerrainNode`
- `TerrainApp.cpp` — инициализация и обновление дерева

**Как работает:**
1. **Инициализация** (`QuadTree::Initialize`): строится дерево с корнем на весь террейн. Узлы делятся до минимального размера (`minNodeSize`) или до максимальной глубины LOD.
2. **Обновление** (`QuadTree::Update`): для каждого узла:
   - проверяется видимость (Frustum Culling);
   - вычисляется LOD по расстоянию до камеры (`CalculateLOD`);
   - решается, рендерить узел или спускаться к детям (`ShouldSubdivide`).
3. **Выбор LOD** (`CalculateLOD`): расстояние от камеры до центра узла сравнивается с порогами `mLODDistances`. Чем ближе — тем меньше номер LOD (больше деталей).
4. **Подразделение** (`ShouldSubdivide`): если камера близко (`distXZ < node->Size * 1.5f`), узел не рендерится, вместо него рисуются дочерние узлы.

**Связь с рендерингом:** В `DrawTerrainNodes()` для каждого видимого узла берётся меш по имени `lod0`–`lod4` (256, 128, 64, 32, 16 вершин по сетке). LOD узла определяет, какой меш рисовать.

### 2. Frustum Culling

**Что это:** Отсечение объектов, которые находятся вне пирамиды видимости камеры (frustum). Невидимые узлы не рисуются.

**Где реализовано:**
- `QuadTree.h` — структура `BoundingBoxAABB` с методом `Intersects`
- `QuadTree.cpp` — `BoundingBoxAABB::Intersects` (проверка AABB vs 6 плоскостей frustum)
- `TerrainApp.cpp` — `ExtractFrustumPlanes`, вызов `QuadTree::Update` с плоскостями

**Как работает:**
1. **Извлечение плоскостей** (`ExtractFrustumPlanes`): из матрицы view-projection получаются 6 плоскостей (Left, Right, Bottom, Top, Near, Far).
2. **AABB узла:** каждый `TerrainNode` имеет `Bounds` (Center + Extents) — выровненный по осям бокс.
3. **Проверка** (`BoundingBoxAABB::Intersects`): для каждой плоскости выбирается «положительная» вершина AABB и проверяется знак расстояния. Если хотя бы для одной плоскости вершина с отрицательной стороны — бокс вне frustum.
4. **Использование:** в `UpdateNode` первым делом вызывается `node->Bounds.Intersects(frustumPlanes)`. Если `false` — узел и его дети не обрабатываются.

### 3. Генерация тайлов с картой высот

**Что это:** Карта высот — текстура, где значение пикселя задаёт высоту точки. Тайлы — части ландшафта с готовыми текстурами (высота, диффуз, нормали).

**Где реализовано:**
- `Terrain.cpp` — загрузка и генерация heightmap
- `TerrainApp.cpp` — загрузка DDS-текстур из `TerrainDetails/003/`
- Папка `TerrainDetails/` — тайлы, сгенерированные внешним инструментом (Gaea)

**Генераторы в проекте:**

| Генератор | Метод | Файл/место |
|-----------|-------|------------|
| **Gaea (внешний)** | Профессиональный софт для террейнов. Создаёт DDS: Height_Out.dds, Weathering_Out.dds, Normals_Out.dds | `TerrainDetails/001`, `002`, `003` |
| **LoadHeightmapDDS** | Загрузка DDS с диска | `Terrain::LoadHeightmapDDS` |
| **LoadHeightmap** | Загрузка raw (8/16 бит) | `Terrain::LoadHeightmap` |
| **GenerateProceduralHeightmap** | Perlin noise с октавами | `Terrain::GenerateProceduralHeightmap` |

**Порядок загрузки** (в `TerrainApp::Initialize`):
1. Пытается загрузить `TerrainDetails/003/Height_Out.dds`.
2. Если не удалось — вызывает `GenerateProceduralHeightmap(256, 256, 4.0f, 6)` (Perlin, 6 октав).

**Perlin noise** (`Terrain.cpp`): используется для процедурной генерации. Таблица перестановок 256→512, функции `Fade`, `Lerp`, `Grad` для сглаживания.

### 4. Рисование кисточкой по цвету террейна (доп. задание)

**Что это:** Режим раскраски ландшафта кистью по месту клика мыши — аналог Landscape Paint в редакторе Unreal. Цвет рисуется поверх диффузной текстуры террейна в мировых координатах; кисть имеет настраиваемый размер и мягкий falloff.

**Где реализовано:**
- `TerrainApp.cpp` — paint-текстура (512×512 RGBA), ray-terrain пересечение, обновление кисти, обработка мыши и клавиш R/G/B, +/-
- `Shaders/Terrain.hlsl` — текстура `gPaintMap` (t3), смешивание диффуза с цветом кисти по альфе в пиксельном шейдере

**Как работает:**
1. **Paint-текстура:** В `BuildDescriptorHeaps()` создаётся текстура 512×512 RGBA в координатах террейна (UV 0–1 по XZ). Данные хранятся в CPU-буфере `mPaintData`; при изменении помечается `mPaintTextureNeedsUpdate`, в `Draw()` выполняется копирование на GPU (Copy Dest → UpdateSubresources → Shader Resource).
2. **Ray casting** (`RayTerrainIntersect`): по экранным координатам мыши строится луч (unproject near/far в мировые), луч шагами пересекается с высотой террейна (`Terrain::GetHeight`), затем бинарный поиск для уточнения точки попадания.
3. **Кисть** (`PaintOnTerrain`): мировая точка переводится в UV террейна и в пиксельные координаты paint-текстуры. В круге радиуса `mBrushSize` (в мировых единицах, пересчитанных в пиксели) применяется мягкий falloff (квадратичный по расстоянию), цвет кисти (`mPaintColor`) смешивается с текущим цветом в `mPaintData` по альфе; флаг `mPaintTextureNeedsUpdate = true`.
4. **Шейдер:** В PS сэмплируются `gDiffuseMap` и `gPaintMap` по одним UV; `albedo = lerp(diffuseColor, paintColor.rgb, paintColor.a)`; при заметной альфе краски — лёгкое усиление для наглядности.

**Управление допом:** ЛКМ — рисование (зажать и вести), ПКМ — поворот камеры. R / G / B — цвет кисти (красный / зелёный / синий). + / − — увеличение / уменьшение размера кисти.

## Связи между компонентами

```
TerrainApp (главный класс)
    │
    ├── Terrain
    │   ├── Heightmap (DDS / raw / procedural)
    │   ├── BuildGeometry() → 5 LOD-мешей (lod0..lod4)
    │   └── GetHeight(), GetNormal() — запросы высоты
    │
    ├── QuadTree
    │   ├── TerrainNode (узлы с Bounds, LODLevel)
    │   ├── Update(camera, frustum) → Frustum Culling + выбор LOD
    │   └── GetVisibleNodes() → список узлов для отрисовки
    │
    └── DrawTerrainNodes()
        └── для каждого узла: ObjectCB (world, texTransform) + DrawIndexed(lod0..lod4)
```

**Поток данных за кадр:**
1. `Update()`: камера → `ExtractFrustumPlanes` → `QuadTree::Update` → `GetVisibleNodes`
2. `UpdatePerObjectCB()`: для каждого видимого узла — world-матрица, UV, LOD
3. `Draw()`: `DrawTerrainNodes()` — для каждого узла DrawIndexed с нужным LOD-мешем

## Шейдеры и карта высот

**Terrain.hlsl:**
- Вершины приходят в локальных координатах (патч -0.5..0.5).
- Высота берётся из `gHeightMap` по UV (`FetchHeightWorld`).
- Нормали считаются в шейдере по центральным разностям в UV (`NormalFromHeight`).
- `gTexTransform` задаёт UV для каждого тайла (масштаб и смещение).
- Пиксельный шейдер: диффуз сэмплируется из `gDiffuseMap`, раскраска кисти — из `gPaintMap` (t3); итоговый albedo = смешивание по альфе краски (как в редакторе ландшафта Unreal).

## Управление

- **WASD** — движение камеры
- **Q/E** — вверх/вниз
- **Shift** — ускорение
- **ЛКМ + движение** — рисование кисточкой по террейну (удерживать и вести)
- **ПКМ + движение** — поворот камеры
- **R / G / B** — цвет кисти (красный / зелёный / синий)
- **+ / −** — размер кисти
- **1** — переключение wireframe (цвет по LOD: красный=0, зелёный=1, синий=2, жёлтый=3, magenta=4)

## Краткие ответы для защиты

**«Как реализовано квадро-дерево для LOD?»**  
Квадро-дерево строится в `QuadTree::BuildTree`. Каждый узел — квадратная область. При обновлении по расстоянию до камеры выбирается LOD и решается, рендерить узел или спускаться к детям. Видимые листья собираются в `GetVisibleNodes` и рисуются с мешами lod0–lod4.

**«Как работает Frustum Culling?»**  
Из view-projection извлекаются 6 плоскостей frustum. У каждого узла есть AABB. Метод `BoundingBoxAABB::Intersects` проверяет пересечение с плоскостями. Если узел вне frustum, он и его дети не рисуются.

**«Как генерируются тайлы с картой высот?»**  
Используются два варианта: 1) внешний генератор Gaea (DDS в `TerrainDetails/`); 2) процедурная генерация Perlin noise в `GenerateProceduralHeightmap`. Карта высот загружается в GPU как текстура и сэмплируется в вершинном шейдере.

**«Как реализовано рисование кисточкой по террейну?»**  
Отдельная paint-текстура 512×512 в UV террейна; при ЛКМ луч из камеры пересекается с высотой террейна (`RayTerrainIntersect`), точка попадания переводится в UV и в пиксели paint-текстуры. В круге заданного радиуса пиксели смешиваются с цветом кисти (мягкий falloff), буфер копируется на GPU в `Draw()`. В шейдере `gPaintMap` смешивается с диффузом по альфе — получается раскраска «как в Unreal».
